Client-side form Validation

  • Making sure all required form fields are filled out and in the correct format is important.
  • It improves the user experience but prevent the delay from submitting a form to the server and have to wait for it to come back as rejected.
  • It is also vital to do the same checks on the server side in order to protect the system from malicious users.
  • Good form validation provides feedback specific to the issues with each field.
  • Why validate?
    • To get the right data in the right format.
    • To protect user data.
    • To protect the application.

    Built-in form validation

  • Built-in form validation is preformed by the browser and is typically more performant than validation done in JavaScript.
  • Built-in features
    • required specifies the field must be filled in before submission.
    • minlength and maxlength Specify the range of the character count in strings.
    • min and max Specify the range of a number value.
    • type indicates the expected form of the data like a email or phone number
    • step indicates how many units a value will change when the field controls are used.
    • patter takes a regular expression to validate the field against.
    • :valid :invalid CSS pseudo-classes allow styling of a field when it passes or fails validation

    JavaScript form validation

  • Forms are also often validated using JavaScript to prove finer control and feedback.
  • The Constraint Validation API contains a set of methods and properties that allow the user to more easily do validation on common form elements.
  • The API makes the following properties available on the elements it supports
    • validationMessage give you the browser message. The setCustomValidity() can be used to allow you to override the standard validation message for the element.
    • validity is a set of properties to help validate different types of data and their range of acceptable entries.
    • willValidate tells you if the element will be validated.

Fetch

  • Fetch is way to send asynchronous HTTP requests.
  • It returns request responses via promises helping to stay away from callback issues.
  • It is an easy API to use in service workers.
  • Fetch that completes the requests and gets a response will always return its Promise, regardless of the HTTP status code
  • Fetch won't send cross-origin cookies without the credentials set in init options
  • The following is a basic fetch call. You call the URL. Then parse the json out of the response. Then do something with the data. 
    fetch('http://example.com/movies.json')
      .then(response => response.json())
      .then(data => console.log(data));
  • There are a number of request options that can be specified in the init object.
    • method - specify GET, POST, DELETE and more
    • mode - no-cors, cors, and same-origin to let the fetch API know if it can talk to other servers or not.
    • cache - many options default, no-cache, reload, force-cache, only-if-cached, for caching a request, refresh a cached request and more.
    • credentials - same-origin, include, omit. This option lets you decide when and if you want to send credentials to the server.
    • headers - many different pieces of information can go here. An example is the Content-Type such as 'application/json'
    • redirect - manual, follow, error - how to handle a response that comes back with a redirect URL.
    • referrerPolicy - no-referrer, *no-referrer-when-downgrade, origin, origin-when-cross-origin, same-origin, strict-origin, strict-origin-when-cross-origin, unsafe-url
    • body - data passed along for requests like POST and PUT. Body data type should match the Content-Type in the headers
