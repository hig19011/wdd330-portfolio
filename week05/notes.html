Testing

  • System Errors - error that occur outside of app code - rare, but unavoidable when they occur
  • Programmer error - error in app code logic
  • User error - user enters invalid data, does an unexpected order of operations or some other interaction issue
  • Exceptions - error that can't be handled correctly by the program - dividing by zero or internet outage
  • Stack trace - sequence of function calls that led up to a specific point, often an exception
  • Warning - a unexpected issue that doesn't stop a program from functioning like an exception does. These can often led to issues later in the program.
  • Avoid setting the code up to create silent failures. These lead to greater problems down the road. Always handle failures, including logging them and notifying the user when appropriate.
  • Use "Strick Mode" to ensure better programming practices. Add 'use strict' to the top line of your code.
  • Modules from ES6 automatically use strict mode, so you don't need to declare it.
  • Linting tools can analyze code for issues and point out error and place for improvement.
  • All JavaScript feature aren't supported by all browsers. Use feature detection by trying to use a feature before running the code that would make use of the feature.

Debugging

  • The Alert function can be used to provide a poor mans break point. It stops the code from running at its usage allowing you a chance to review the application state.
  • The console is where we can log output for debugging purposes, i.e. console.log(`I am ${age} years old`); to display the value of $age
  • The method call console.trace() will display the current stacktrace in the console.
  • Most browser have a debugger where you can step through your JavaScript code as it runs
  • The debuggers allow you to set a breakpoint where you can pause the program execution. This allows you to see what is happening in your code at that point, line by line.
  • The keyword "debugger" will force a breakpoint in the code. Always remove this keyword before releasing your code for use.
  • The "Error" object can be created by the browser or the application and will at least contain the name of the error, message about the error and a stacktrace of when the error occurred.
  • There are many typed Error object used for specific issues.
  • Error object are thrown when Exceptions occur.
  • Exceptions are handled using try, catch, and finally blocks of code.
  • The try block holds the code that might cause and exception.
  • The catch block holds the code that deal with an exception if it occurs.
  • The finally block is optional, if included will always run its code regardless if an exception was thrown or not. It is good for clean up used resources

Tests

  • Tests can identify errors early in development process.
  • Tests can help reduce the brittleness of the code.
  • Test-driven Development or TTD
    • This process involves writing test before writing the code.
    • The test should always fail, because there isn't any code to support them yet.
    • Then you write the code the test will use until the test passes.
    • Later when you refactor the code, the test should either pass or need refactoring to match.
    • Repeat
    • This process is referred to as "red-green-refactor"
    • Use judiciously, it isn't appropriate for every scenario.
  • There are a number of robust Testing Frameworks.
  • Jest is a common one made by Facebook and is used in this weeks examples.
